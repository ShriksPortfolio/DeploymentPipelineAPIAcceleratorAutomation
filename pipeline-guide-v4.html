API Call Sequence
Phase 1 — Authentication
POST/oauth2/v2.0/token
Acquires bearer token via client_credentials grant using service principal.
Phase 2 — Pipeline Setup
GET/pipelines
Discovers existing pipeline by display name, or creates a new one.
Phase 2 — Stage Assignment
GET/pipelines/{id}/stages
Validates Dev (Stage 0) and UAT (Stage 1) workspace assignments.
Phase 3 — Pre-Deployment Validation
GET/pipelines/{id}/stages/0/artifacts
Lists all Dev artifacts and cross-references against deployment manifest.
Phase 4 — UAT State Check
GET/pipelines/{id}/stages/1/artifacts
Detects if UAT already has items → determines report-only vs full deploy.
Phase 4 — Deploy
POST/pipelines/{id}/deploy
Triggers async deployment. Returns operation ID for polling.
Phase 5 — Poll
GET/pipelines/{id}/operations/{opId}
Polls every 5s until Succeeded, Failed, or 300s timeout.
Phase 6 — TakeOver (first deploy only)
POST/datasets/{id}/Default.TakeOver
Service principal takes ownership of UAT semantic model.
Phase 6 — Bind Gateway (first deploy only)
POST/datasets/{id}/Default.BindToGateway
Binds semantic model to cloud connection gateway and datasource.
Phase 7 — Verification
GET/pipelines/{id}/stages/1/artifacts
Confirms datasets and reports are present in UAT after deployment.
Phase 8 — Audit Log
GET/pipelines/{id}/operations
Extracts full deployment history into structured DataFrame.
Before You Begin
Pre-Setup Requirements
Everything that needs to be in place before the notebook can run. Most items are one-time setup — once configured, all future deployments are fully automated.
🔐
Service Principal in Entra ID
Register an App Registration in Microsoft Entra ID (Azure AD). Generate a client secret and note the Tenant ID, Client ID, and Client Secret. This identity authenticates all API calls.
Required
🏢
Fabric Workspace Permissions
The service principal must be added as a Member (or Admin) to both the Dev and UAT workspaces. Without this, the API calls to list artifacts and trigger deployments will fail with 401/403.
Required
⚙️
Tenant Admin: API Access for SPs
In the Fabric Admin Portal, enable "Allow service principals to use Power BI APIs" under Tenant Settings → Developer Settings. This is a tenant-level toggle that must be on.
Required
🔗
Deployment Pipeline Created
The notebook can auto-create and assign workspaces to the pipeline. However, if your organisation restricts pipeline creation, you may need to pre-create it in the Fabric portal.
Auto-Created
☁️
Cloud Connection (Gateway)
A shareable cloud connection (e.g. LH_Marketing_CRM) must exist in the Fabric portal. The UAT semantic model will bind to this connection's gateway and datasource IDs after first deployment.
Required
🗺️
Manual Connection Mapping
After the first full deployment, you must manually map the UAT semantic model to the named cloud connection in the portal (Settings → Gateway → Maps To). This is a one-time step — the API cannot do this.

01

Authentication

Acquires an OAuth2 bearer token from Entra ID using service principal client_credentials flow. Token scoped to the Power BI API resource. Fails fast if credentials have expired.

POST /oauth2/v2.0/token client_credentials
▶ View Notebook Code
Token Request Body
Constructs the OAuth2 payload using the service principal credentials defined in Cell 1. The scope targets the Power BI API resource with the .default suffix, which requests all statically configured permissions.
body = {
    'client_id':     CLIENT_ID,
    'scope':         f'{RESOURCE}/.default',
    'client_secret': CLIENT_SECRET,
    'grant_type':    'client_credentials'
}
POST to Entra ID Token Endpoint
Sends the credentials to the Microsoft identity platform. If the response is not 200, the notebook halts immediately with a clear error message — most commonly caused by an expired client secret in Entra ID.
response = requests.post(AUTHORITY, data=body,
    headers={'Content-Type': 'application/x-www-form-urlencoded'})

if response.status_code != 200:
    raise Exception("Authentication failed — check client secret expiry")

access_token = response.json()['access_token']
api_headers = {
    'Authorization': f'Bearer {access_token}',
    'Content-Type':  'application/json'
}

Pipeline Setup

Discovers or creates the deployment pipeline by name. Validates that Dev and UAT workspaces are correctly assigned to Stage 0 and Stage 1 respectively. Auto-assigns if missing.

GET /pipelines GET /stages auto-assign workspaces
▶ View Notebook Code
Pipeline Discovery
Queries all deployment pipelines the service principal has access to, then searches for a match by display name. If found, reuses it. If not, creates a new one via POST — making the notebook self-provisioning.
response = requests.get(f"{BASE_URL}/pipelines", headers=api_headers)

pipeline_id = None
for p in response.json().get('value', []):
    if p['displayName'] == PIPELINE_NAME:
        pipeline_id = p['id']
        break

# Create pipeline if it does not exist
if not pipeline_id:
    response = requests.post(
        f"{BASE_URL}/pipelines", headers=api_headers,
        json={"displayName": PIPELINE_NAME})
    pipeline_id = response.json()['id']
Stage Workspace Assignment
Checks each pipeline stage for an assigned workspace. If a stage is unassigned but has a mapped workspace in the configuration, it auto-assigns it. This ensures the pipeline is ready to deploy without any manual portal setup.
workspace_map = {0: DEV_WORKSPACE_ID, 1: UAT_WORKSPACE_ID}

for stage in stages:
    order = stage['order']
    ws_id = stage.get('workspaceId')

    if ws_id:
        print(f"  Stage {order}: {ws_id}")
    elif order in workspace_map:
        requests.post(
            f"{BASE_URL}/pipelines/{pipeline_id}/stages/{order}/assignWorkspace",
            headers=api_headers,
            json={"workspaceId": workspace_map[order]})

03

Pre-Deployment Check

Cross-references every item in the deployment manifest against artifacts in the Dev stage. If any sourceId is missing, the pipeline halts immediately — zero risk of deploying non-existent items.

GET /stages/0/artifacts manifest validation
▶ View Notebook Code
Artifact Discovery and Manifest Validation
Retrieves all artifacts from the Dev stage (datasets, reports, dashboards, dataflows), then validates every item in the ITEMS_TO_DEPLOY dictionary exists in the source. If any ID is missing, an exception halts execution before any changes are made to UAT.
artifacts_url = f"{BASE_URL}/pipelines/{pipeline_id}/stages/0/artifacts"
response = requests.get(artifacts_url, headers=api_headers)
dev_data = response.json()

# Build list of all artifact IDs in the Dev stage
all_stage_ids = []
for item_type in ['datasets', 'reports']:
    all_stage_ids.extend([i['artifactId'] for i in dev_data.get(item_type, [])])

# Cross-reference every deployment item against Dev
all_valid = True
for item_type, items in ITEMS_TO_DEPLOY.items():
    for item in items:
        if item['sourceId'] in all_stage_ids:
            print(f"VALID   {item_type}: {item['sourceId']}")
        else:
            print(f"MISSING {item_type}: {item['sourceId']}")
            all_valid = False

if not all_valid:
    raise Exception("Some items not found in Dev stage")

04

Deploy (Smart Detection)

Checks if UAT already has items. If yes → report-only deployment (preserves existing data connection). If empty → full deployment (semantic model + report). Eliminates the most common deployment error.

POST /deploy report-only vs full async HTTP 202
▶ View Notebook Code
Smart Mode Detection
Queries the UAT stage to check whether datasets and reports already exist. If UAT has items, it switches to report-only mode — this is the key logic that preserves the UAT semantic model's cloud connection mapping and eliminates the need for manual reconfiguration.
# Check if UAT already has items
uat_check = requests.get(uat_artifacts_url, headers=api_headers)
uat_data = uat_check.json()

uat_has_datasets = len(uat_data.get('datasets', [])) > 0
uat_has_reports  = len(uat_data.get('reports', []))  > 0

# Decide deployment mode
if uat_has_datasets and uat_has_reports:
    deploy_mode = "REPORT ONLY"
    deploy_items = {"reports": ITEMS_TO_DEPLOY.get("reports", [])}
else:
    deploy_mode = "FULL (Dataset + Report)"
    deploy_items = {
        "datasets": ITEMS_TO_DEPLOY.get("datasets", []),
        "reports":  ITEMS_TO_DEPLOY.get("reports", [])
    }
Deploy Request
Constructs and sends the deployment payload. The API returns HTTP 202 (Accepted) with an operation ID — the deployment runs asynchronously and is monitored in Phase 5. Options include allowTakeover for ownership transfer between service principals.
deploy_body = {
    "sourceStageOrder": SOURCE_STAGE_ORDER,
    "options": {
        "allowCreateArtifact":    ALLOW_CREATE,
        "allowOverwriteArtifact": ALLOW_OVERWRITE,
        "allowTakeover":          True
    },
    "note": f"Automated deployment ({deploy_mode})"
}

response = requests.post(
    f"{BASE_URL}/pipelines/{pipeline_id}/deploy",
    headers=api_headers, json=deploy_body)

operation_id = response.json()['id']

05

Poll for Completion

Async polling loop monitors the deployment operation every 5 seconds, with a 300-second timeout. Extracts execution plan steps on success, or detailed error codes on failure.

GET /operations/{opId} 5s interval 300s max
▶ View Notebook Code
Asynchronous Polling Loop
The Power BI deploy endpoint is asynchronous — it returns immediately with an operation ID. This loop polls the operation status endpoint every 5 seconds until it reaches a terminal state (Succeeded or Failed). On success, it extracts the execution plan details. On failure, it surfaces the error code and details for debugging.
operation_url = f"{BASE_URL}/pipelines/{pipeline_id}/operations/{operation_id}"
max_wait      = 300   # 5-minute timeout
poll_interval = 5
elapsed       = 0

while elapsed < max_wait:
    response = requests.get(operation_url, headers=api_headers)
    operation = response.json()
    status    = operation.get('status', 'Unknown')

    if status == 'Succeeded':
        # Extract execution plan steps
        steps = operation.get('executionPlan', {}).get('steps', [])
        for step in steps:
            src = step.get('sourceAndTarget', {})
            print(f"  {src.get('sourceDisplayName')} — {step.get('status')}")
        break

    elif status == 'Failed':
        raise Exception("Deployment failed")

    time.sleep(poll_interval)
    elapsed += poll_interval

06

Auto-Configuration

First deploy only: service principal takes over the UAT semantic model, then binds it to the cloud connection gateway. Skipped entirely on subsequent report-only deployments.

POST /TakeOver POST /BindToGateway first deploy only
▶ View Notebook Code
Conditional Execution — Skip on Report-Only
The is_first_deployment flag is set in Phase 4. If this is a report-only deployment, the entire auto-configuration is skipped because the UAT semantic model already has its connection configured from a previous full deployment.
if not is_first_deployment:
    print("SKIPPED — report-only deployment, connection already configured")
else:
    # Step 1: Discover UAT dataset ID from pipeline artifacts
    uat_dataset_id = uat_datasets[0]['artifactId']

    # Step 2: Service principal takes over the semantic model
    takeover_url = f"{BASE_URL}/groups/{UAT_WORKSPACE_ID}/datasets/{uat_dataset_id}/Default.TakeOver"
    requests.post(takeover_url, headers=api_headers)

    # Step 3: Bind to cloud connection
    bind_url = f"{BASE_URL}/groups/{UAT_WORKSPACE_ID}/datasets/{uat_dataset_id}/Default.BindToGateway"
    bind_body = {
        "gatewayObjectId":    UAT_GATEWAY_ID,
        "datasourceObjectIds": [UAT_DATASOURCE_ID]
    }
    requests.post(bind_url, headers=api_headers, json=bind_body)

Verification

Queries the UAT stage and confirms all expected datasets and reports are present with correct names and IDs. Acts as the automated quality gate before the pipeline completes.

GET /stages/1/artifacts quality gate
▶ View Notebook Code
UAT Artifact Verification
After deployment completes, this phase queries the UAT stage directly to confirm the expected artifacts are present. It iterates through datasets and reports, printing each artifact's display name and ID. This acts as the automated quality gate — visual confirmation that deployment succeeded at the artifact level, not just the operation level.
artifacts_url = f"{BASE_URL}/pipelines/{pipeline_id}/stages/1/artifacts"
response = requests.get(artifacts_url, headers=api_headers)

if response.status_code == 200:
    data = response.json()
    for item_type in ['datasets', 'reports']:
        items = data.get(item_type, [])
        if items:
            print(f"  {item_type.upper()} now in UAT:")
            for item in items:
                print(f"    {item.get('artifactDisplayName')} | {item.get('artifactId')}")

08

Deployment Log

Extracts the complete deployment history into a structured pandas DataFrame. Captures operation IDs, timestamps, item names, step statuses, error details, and deployment notes — full audit trail.

GET /operations pandas DataFrame audit trail
▶ View Notebook Code
Operations History Extraction
Retrieves every deployment operation from the pipeline, iterates through each operation's execution plan steps, and builds a flat list of log records. Each record captures the operation ID, status, timestamps, item names and types, source/target IDs, error details (if any), and the deployment note.
operations_url = f"{BASE_URL}/pipelines/{pipeline_id}/operations"
response = requests.get(operations_url, headers=api_headers)
operations = response.json().get('value', [])

log_data = []
for op in operations:
    steps = op.get('executionPlan', {}).get('steps', [])
    for step in steps:
        src_target = step.get('sourceAndTarget', {})
        log_data.append({
            'Operation ID':   op.get('id'),
            'Status':         op.get('status'),
            'Started':        op.get('executionStartTime'),
            'Ended':          op.get('executionEndTime'),
            'Item Name':      src_target.get('sourceDisplayName'),
            'Item Type':      step.get('type'),
            'Step Status':    step.get('status'),
            'Error Code':     step.get('error', {}).get('errorCode', '')
        })

df_log = pd.DataFrame(log_data)

Deployment Intelligence
The pipeline automatically selects the optimal deployment strategy based on the current state of the UAT workspace.

existing items?

Report Only Preserves UAT connection
No
Full Deploy Dataset + Report + Config

Why this matters: When a semantic model is redeployed to UAT, its cloud connection mapping is lost and must be manually reconfigured. By detecting that UAT already has the semantic model and deploying only the report, the pipeline preserves the existing connection — making every subsequent deployment fully automated with zero manual intervention.
